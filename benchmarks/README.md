# Tools

## sysbench

sysbench is a scriptable multi-threaded benchmark tool based on LuaJIT. It is most frequently used for database benchmarks, but can also be used to create arbitrarily complex workloads that do not involve a database server.

sysbench comes with the following bundled benchmarks:
- oltp_*.lua: a collection of OLTP-like database benchmarks
- fileio: a filesystem-level benchmark
- cpu: a simple CPU benchmark
- memory: a memory access benchmark
- threads: a thread-based scheduler benchmark
- mutex: a POSIX mutex benchmark

[sysbench - Github](https://github.com/akopytov/sysbench)

## Hackbench

hackbench is a benchmark and a stress test for the kernel scheduler that is part of the rt-tests suite. Hackbench also stresses parts of the memory subsystem through repeated setup and teardown of threads. Additionally, it stresses inter-process communication (e.g. local sockets, pipes) to a certain degree.

The purpose of the tool is to help identify bottlenecks in the system that should be eliminated or optimized.

Hackbench can be used to generate a system load while measuring latencies with Cyclictest. However, it cannot be used to accurately simulate the load generated by any given RT application because the program does not test communication with devices.

<https://wiki.linuxfoundation.org/realtime/documentation/howto/tools/hackbench>

## Cyclictest

Cyclictest accurately and repeatedly measures the difference between a thread's intended wake-up time and the time at which it actually wakes up from clock_nanosleep in order to provide statistics about the system's latencies. It can measure latencies in real-time systems caused by the hardware, the firmware, and the operating system.

Cyclictest is most commonly used for benchmarking RT systems. It is one of the most frequently used tools for evaluating the relative performance of real-time systems.

<https://wiki.linuxfoundation.org/realtime/documentation/howto/tools/cyclictest/start>

## perf-sched

perf-sched - Tool to trace/measure scheduler properties (latencies)

[perf-sched(1) — Linux manual page](https://www.man7.org/linux/man-pages/man1/perf-sched.1.html)

# Benchmarking scenerios
- For schedulers:
    - Benchmark all schedulers under specific workloads and conditions on different machines (automated with Bash script).
- For scx-adapt configurations:
    - Create scx-adapt configurations considering benchmark results and scheduler explanations.
    - Create mixed-workload-benchmarks then benchmark system scheduler and scx-adapt configurations.


# System setup
- A minimal Linux distribution with unnecessary services disabled (to prevent benchmark contamination)
- **Disabled frequency scaling**: `cpupower frequency-set -g performance`
- **Disabled turbo mode**:
    - Intel: `echo 1 > /sys/devices/system/cpu/intel_pstate/no_turbo`
    - AMD: `echo 0 > /sys/devices/system/cpu/cpufreq/boost`
- Verify: `cpupower frequency-info`
- **Core isolation (cgroup2 - systemd)**: [Splitting and isolating CPU cores in Linux](https://medium.com/@shahriar.rezghi.sh/splitting-and-isolating-cpu-cores-in-linux-ad6ac4dc1904)
    - GRUB parameters for kernel: `kthread_cpus=16-31 irqaffinity=16-31 nohz_full=0-15 rcu_nocbs=0-15 rcu_nocb_poll`
